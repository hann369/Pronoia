- CTA: Primär = "Protokoll starten" / Sekundär = "Batch #001 ansehen"

### Technical Specs

```
Scroll:       GSAP ScrollTrigger + Lenis smooth scroll
Lazy Load:    IntersectionObserver (native) für alle Bilder unterhalb Fold
Reduced Motion: prefers-reduced-motion deaktiviert WebGL/GSAP automatisch
Mobile:       Three.js Grass/Particles deaktiviert → statischer Gradient-Fallback
Performance:  LCP < 2s, Total Assets < 3MB, CLS < 0.1
Internationalisierung: DE/EN via navigator.language + IP-Geo-Fallback
Image Format: WebP primär, AVIF mit <picture>-Fallback
Critical CSS: Inline im <head>. Restliches CSS async geladen.
```

> ⚠️ **Antigravity-Fehlerprävention (Performance):**
> - Three.js-Instanz MUSS in `requestIdleCallback` initialisiert werden, nicht im main thread.
> - GSAP ScrollTrigger `kill()` beim Verlassen der Route/Component — sonst Memory Leak bei SPA.
> - Lenis scroll + GSAP: `lenis.on('scroll', ScrollTrigger.update)` verbinden, sonst Desync.
> - `prefers-reduced-motion` in JS prüfen: `window.matchMedia('(prefers-reduced-motion: reduce)').matches`

## 4. PLATFORM & EXECUTION SYSTEM (Post-Signup)

### Core Philosophy

> Protocol-Runner, nicht Dashboard. Die Plattform ist ein unsichtbarer Manager. Sie zeigt nie alles. Sie zeigt immer nur: **What's next?**

### User Flow

```
Onboarding (Stage-Assessment: 5 Fragen max.)
    ↓
Protocol Router → generiert 3–5 Next Actions (Stage + HRV + circadiane Phase)
    ↓
Focus Block startet (Ultradian Timer 90/20)
    ↓
Friction Log (1-Tap: ✅ Glatt / ⚠️ Reibung / ❌ Abbruch + 1-Wort Grund)
    ↓
Post-Block Reflection (60s max. Prompt)
    ↓
Protocol Router aktualisiert Queue
```

### UI Components

| Modul | Funktion | Regel |
|---|---|---|
| **Protocol Router** | LLM + Rule-Engine generiert 3–5 Next Actions basierend auf Stage, HRV, circadianer Phase | Nur 3 sichtbar. Rest im Queue. Kein "Alle anzeigen"-Button. |
| **Ultradian Timer** | 90/20 Timer mit Breath-Cue, Ambient-Sound, Post-Block Reflection | Kein manueller Start. Triggered by Protocol. |
| **Friction Log** | 1-Tap: ✅ / ⚠️ / ❌ + 1-Wort Grund | Kein Score. Nur Debug-Daten. Kein Verlauf sichtbar. |
| **Contextual Nudges** | Push/In-App: HRV ↓ → 10 Min NSDR · 15:30 → Prep Post-Gym Fuel | Just-in-Time Adaptive Intervention (JITAI) |
| **Micro-Protocol Library** | 3–7 Min Videos: Ausführung, keine Theorie | Mirror-Neuron Activation. Kein autoplay. |

### Data Architecture

```
Firebase Firestore (User State, Protocol Queue, Friction Logs)
    ↓
Groq API (LLM Router — Background only, nie sichtbar im UI)
    ↓
Supabase (Biomarker Sync, relationale Abfragen — Phase 2+)
    ↓
LocalStorage + Service Worker (Offline-First, sync on reconnect)
```

**Firestore Collection Schema:**

```json
users/{uid} {
  stage:         "1" | "2" | "3" | "4",
  circadian:     { phase: "morning" | "afternoon" | "evening", updated: timestamp },
  hrv:           { value: number, trend: "up" | "down" | "stable", updated: timestamp },
  protocol:      { active: string, queue: [string], lastCompleted: timestamp },
  frictionLog:   [{ blockId: string, result: "smooth" | "friction" | "abort", tag: string, ts: timestamp }],
  replenishment: { items: [{ productId: string, daysLeft: number, autoship: boolean }] }
}
```

> ⚠️ **Antigravity-Fehlerprävention (Firestore):**
> - Firestore Security Rules IMMER auf `request.auth.uid == resource.data.owner` prüfen. Standard-Rules sind offen.
> - Kein `collection.get()` ohne Limit — führt zu unkontrollierten Reads und Kosten-Explosion.
> - Offline-First: Firestore `enablePersistence()` aufrufen vor dem ersten `onSnapshot`.
> - HRV-Daten sind Gesundheitsdaten (GDPR Art. 9). Explizit als `sensitiveData: true` taggen und separate Einwilligungsstrecke einbauen.

### UX Rules

```
✅ Zero-Clutter: Kein Menü während aktiver Focus-Blocks
✅ Context-First: UI passt sich an Tageszeit/HRV-Trend an (nicht an "Mood")
✅ Offline-First: LocalStorage + Service Worker, sync on reconnect
✅ Privacy-Default: Kein PII gespeichert. Anonymized metrics. Opt-in für Sync.
✅ No loading spinners für Aktionen < 300ms (optimistic UI)
❌ Kein "Zurück"-Button während Focus-Block aktiv
❌ Keine Push-Notifications ohne explizite Kategorie-Einwilligung
```

## 5. PROTOCOL-INTEGRATED REPLENISHMENT ENGINE

### Philosophie

> Kein "Shop". Kein "Add to Cart". Replenishment ist Teil des Protokolls. Der Nutzer interagiert damit nur, wenn der Stack es benötigt.

### UX Flow

```
Stack-Tracking (Einmalige Eingabe: Startdatum + Packungsgröße)
    ↓
7-Tage-Warning: In-App Nudge + E-Mail (kein Push by default)
    ↓
Auto-Replenish ausgelöst ODER manueller Review (skip / pause / anpassen)
    ↓
Stripe Subscription Charge + Versandbestätigung
    ↓
Neue Charge eingetragen → Stack-Timer resettet
```

### Trigger-Logic

| Trigger | Action |
|---|---|
| Stock ≤ 7 Tage | In-App Nudge + E-Mail |
| Nutzer skippt > 2× | Auto-Replenish pausiert, manuelle Review-Anfrage |
| HRV ↓ über 14 Tage | Suggest Glycin / Mg Dosiserhöhung |
| Stage-Upgrade | Recommend Protocol-Modifier (Clothing, Filter) |
| Charge-Wechsel | Neue CoA automatisch verlinkt vor Versand |

### Product Categories

| Kategorie | Beispiele | Positioning |
|---|---|---|
| **Labs** | PX-V1, Bromantane, Electrolytes | Kühl, klinisch, CoA-basiert, batch-limitiert |
| **Health** | Baumwolle/Leinen, Aleppo-Seife, Luftfilter | Warm, natürlich, Environment-Modifier |
| **Education** | Protokoll-Guides, Circadian Deep-Dives | PDF/Video, einmalig oder Bundle |

### Compliance & Safety

```
✅ EU Nootropics Regulation: Bromantane als "Research Chemical" klar deklariert
✅ CoA vor jeder Charge öffentlich (PDF + QR auf Verpackung)
✅ Stripe Subscriptions: cancel anytime / skip / pause — Buttons IMMER prominent
✅ Keine Medical Claims. Nur "Protocol Integration" & "Support"
✅ AGB-Klausel: Nutzer bestätigt Research-Chemical-Status vor Kauf
❌ Keine automatische Charge ohne explizite Wiederholungs-Einwilligung (GDPR + PSD2)
```

> ⚠️ **Antigravity-Fehlerprävention (Stripe):**
> - Stripe Webhook-Signatur IMMER verifizieren (`stripe.webhooks.constructEvent`). Unverified Webhooks sind ein kritisches Sicherheitsleck.
> - `cancel_at_period_end` für Subscriptions verwenden, nicht sofortiges Cancel — Nutzer verliert sonst bereits bezahlten Zugang.
> - Idempotency Keys bei Payment-Intents setzen, um Doppelbelastungen bei Retry zu verhindern.
> - Stripe im Test-Modus entwickeln. Keine Live-Keys in `.env.development`.

## 6. TECH STACK & DEVELOPMENT SPECS

### Frontend

| Layer | Tech | Notes |
|---|---|---|
| **Web (Landing + Platform)** | Vanilla HTML/CSS/JS + GSAP + Three.js + Lenis | Kein Framework-Overhead für Landing Page |
| **Android App** | Kotlin + Jetpack Compose + Firebase + Hilt + WorkManager | Protocol Runner nativ |
| **PWA** | Service Worker + Cache API + IndexedDB | Offline-First Protocol Runner |

> ⚠️ **Antigravity-Fehlerprävention (Frontend):**
> - Three.js auf mobilen Geräten: Immer `renderer.setPixelRatio(Math.min(window.devicePixelRatio, 2))` setzen — höhere Pixel Ratios killen Performance auf Mid-Range Geräten.
> - Service Worker Update-Flow implementieren: `skipWaiting()` + `clients.claim()` + User-Prompt für Reload. Ohne das läuft ein veralteter SW still weiter.
> - IndexedDB für Offline-Daten. `localStorage` hat 5MB Limit und ist synchron (blockiert Main Thread).

✅ Rate Limiting auf allen API Routes (Vercel: @upstash/ratelimit empfohlen)
